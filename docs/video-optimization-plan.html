一、当前管线分析

1.1 数据流

当前视频数据流经 7 个阶段，每阶段都有一定的延迟开销：

阶段组件估算延迟当前状态
1. 编码Mac (H.264/HEVC 硬件编码)~10-20ms无法控制（发送端）
2. 传输RTP over UDP (WiFi)~5-20ms取决于网络质量
3. 接收+解密UxPlay (C, RTP线程)~2-5ms已优化（非阻塞帧池）
4. 帧拷贝memcpy → DirectByteBuffer~0.3-0.5ms可优化（zero-copy）
5. JNI 传递Java onVideoData 回调~0.1ms已优化（DirectByteBuffer）
6. 硬件解码MediaCodec async~5-15ms部分优化（KEY_LOW_LATENCY）
7. 渲染SurfaceView → SurfaceFlinger~3-16ms未优化（无帧步调控制）

端到端估算总延迟：25-76ms（约 1.5-4.5 帧 @60fps）。其中可通过软件优化降低的部分主要是阶段 4-7，理论可降低 5-15ms。

1.2 已有优化

  • DirectByteBuffer 帧池（8×4MB）— 消除每帧 GC 压力
  • sem_trywait 非阻塞获取 — 防止 RTP 线程阻塞
  • MediaCodec async 回调模式 — 避免同步 dequeue 轮询
  • KEY_LOW_LATENCY=1（API 30+）— 解码器低延迟模式
  • 最新帧优先策略 — 实时流场景减少累积延迟
  • NAL 偏移引用解析 — 减少 ByteArray 拷贝

1.3 缺失的优化

  • MediaFormat 键不完整：缺少 KEY_PRIORITY、KEY_OPERATING_RATE、KEY_FRAME_RATE、KEY_COLOR_STANDARD、KEY_COLOR_RANGE、KEY_COLOR_TRANSFER
  • 无色彩空间元数据：解码器不知道输入流的色彩标准，可能使用错误的色彩转换矩阵
  • 无帧步调控制：releaseOutputBuffer(idx, true) 立即渲染，不与 vsync 对齐
  • 单帧 pending：只缓冲一帧，高频场景下丢帧率高
  • 无自适应播放：分辨率变化时需要完全重建 codec，无法无缝切换
  • 无渲染时间戳：未使用 releaseOutputBuffer(idx, timestamp) API 进行音视频同步

二、优化方案

2.1 延迟优化

优化 1: 补全 MediaFormat 解码器配置键

低风险 API 23+ 影响：高

当前 configureCodec() 只设置了 KEY_LOW_LATENCY 和 KEY_MAX_INPUT_SIZE。缺少以下关键键：

键API Level建议值作用
KEY_PRIORITY230 (实时)告知解码器这是实时流，优先调度
KEY_OPERATING_RATE2360 或 120提示解码器目标帧率，优化内部流水线
KEY_FRAME_RATE2160输入流帧率信息，帮助解码器调度
// 在 configureCodec() 中添加:
format.setInteger(MediaFormat.KEY_PRIORITY, 0)  // 0 = realtime
format.setInteger(MediaFormat.KEY_OPERATING_RATE, 120)  // 目标 fps × 2 余量
format.setInteger(MediaFormat.KEY_FRAME_RATE, 60)

可行性：极高。纯 API 调用，不改变管线结构，设备 API 24 已支持 KEY_PRIORITY 和 KEY_OPERATING_RATE。

预期收益：解码器在硬件层面更积极地调度，减少内部缓冲延迟 2-5ms。

优化 2: 使用 releaseOutputBuffer 渲染时间戳

中风险 API 21+ 影响：中

当前 onOutputBufferAvailable 使用 releaseOutputBuffer(index, true) 立即渲染。Android API 21+ 支持 releaseOutputBuffer(index, renderTimestampNs)，可以指定渲染时间戳。

// 当前:
codec.releaseOutputBuffer(index, true)

// 优化为: 使用 NTP 时间戳对齐 vsync
val renderNs = System.nanoTime()  // 或基于 pts 计算
codec.releaseOutputBuffer(index, renderNs)

可行性：中等。需要将 NTP 时间戳转换为 nanotime 基准，且并非所有设备都精确支持 timestamp 渲染。如果时间戳计算有误，可能导致画面卡顿。

预期收益：帧渲染与 vsync 对齐，减少画面撕裂和抖动，视觉流畅度提升。

风险：时间戳映射不准会导致帧延迟增加而非减少。建议先实现但不使用时间戳（传 0），作为后续调优选项。

优化 3: 多帧 pending 缓冲（2-3 帧）

中风险 影响：中

当前只缓冲 1 帧（最新帧覆盖旧帧）。在解码器短暂卡顿时（如处理 B 帧或 IDR 帧），后续帧会被丢弃。改为 2-3 帧环形缓冲可减少丢帧率。

// 当前: 单帧 pendingFrame
// 优化为: ArrayDeque 帧队列（最大 3 帧）
private val pendingFrames = ArrayDeque()
data class FrameData(val buffer: ByteBuffer, val size: Int, val pts: Long, val isH265: Boolean)

fun handleFrame(...) {
    pendingFrames.addLast(FrameData(data, size, ntpTime, isH265))
    if (pendingFrames.size > MAX_PENDING) {
        pendingFrames.removeFirst()  // 丢弃最旧帧
    }
    tryFeedPendingFrame()
}

权衡：增加最多 2 帧延迟（~33ms @60fps），但减少丢帧率。对于投屏场景，延迟增加不明显但流畅度提升明显。

可行性：中等。需要修改 feedFrameToCodec 逻辑和缓冲区归还时机。

2.2 色彩与显示质量优化

优化 4: 设置色彩空间元数据

低风险 API 24+ 影响：高

当前完全没有色彩空间元数据。AirPlay 屏幕镜像使用 BT.709 色彩标准（SDR 8-bit），但解码器不知道这一点，可能使用错误的 YUV→RGB 转换矩阵，导致色彩偏移。

键建议值含义
KEY_COLOR_STANDARD1 (BT.709)HDR 内容使用 4 (BT.2020)
KEY_COLOR_RANGE2 (完整范围)AirPlay 通常使用 0-255 完整范围
KEY_COLOR_TRANSFER6 (BT.709)SDR 使用 BT.709 传输函数
// 在 configureCodec() 中添加:
format.setInteger(MediaFormat.KEY_COLOR_STANDARD,
    MediaFormat.COLOR_STANDARD_BT709)  // = 1
format.setInteger(MediaFormat.KEY_COLOR_RANGE,
    MediaFormat.COLOR_RANGE_FULL)  // = 2
format.setInteger(MediaFormat.KEY_COLOR_TRANSFER,
    MediaFormat.COLOR_TRANSFER_BT709)  // = 6

可行性：极高。纯 API 调用，设备 API 24 已支持。从 AirPlay 规范可知 macOS 屏幕镜像使用 H.264 High Profile + BT.709。

预期收益：纠正色彩转换矩阵，消除色彩偏移/发白/发灰问题。这是视觉质量提升最明显的单项优化。

优化 5: SPS 解析提取色彩信息

中风险 影响：中

H.264 SPS NAL 单元中包含 vui_parameters 字段，其中有 colour_primaries、transfer_characteristics、matrix_coefficients 信息。可以从 SPS 中解析这些值，动态设置色彩标准。

// H.264 SPS VUI 参数解析 (简化):
// - matrix_coefficients = 1 → BT.709
// - matrix_coefficients = 9 → BT.2020
// - colour_primaries = 1 → BT.709
// - transfer_characteristics = 1 → BT.709
//
// 需要扩展 extractCsdOptimized() 返回色彩信息
// 然后在 configureCodec() 中使用实际值

可行性：中等。需要实现 SPS VUI 解析器（~50 行 C/位操作代码），但一旦实现可自动适配 HDR 内容。

预期收益：自动适配不同色彩空间，未来支持 HDR 投屏时不需要手动修改。

优化 6: 10-bit / HDR 支持

高风险 API 24+ 影响：低（当前）

macOS 屏幕镜像目前只发送 8-bit SDR H.264 流，HDR 暂不可用。但未来 Apple 可能推送 10-bit HEVC HDR 流。可预留支持：

  • HEVC Main 10 Profile: HEVCProfileMain10
  • HDR 静态元数据: KEY_HDR_STATIC_INFO（SMPTE ST.2086）
  • 色彩标准: COLOR_STANDARD_BT2020
  • 色彩传输: COLOR_TRANSFER_HLG 或 COLOR_TRANSFER_ST2084

可行性：低（当前不可用，AirPlay 不发 HDR 流）。仅作为未来预留设计，当前不实现。

2.3 帧率优化

优化 7: SurfaceView setFixedSize 优化

低风险 影响：中

当前 SurfaceView 使用 match_parent 全屏铺满。当视频分辨率与屏幕宽高比不一致时，SurfaceFlinger 需要进行额外的缩放合成。设置 setFixedSize 使 Surface 缓冲区与视频分辨率匹配，让 GPU 而非 SurfaceFlinger 进行缩放。

// 在 surfaceChanged 或 onVideoSize 回调中:
holder.setFixedSize(videoWidth, videoHeight)
// SurfaceView 会自动按 videoWidth:videoHeight 比例缩放到屏幕

权衡：如果视频分辨率与屏幕分辨率完全一致（自适应模式），setFixedSize 无效果甚至可能降低质量。仅在手动分辨率模式下有意义。

可行性：高。但需注意 setFixedSize 会触发 surfaceChanged 回调，需要避免循环。

优化 8: 自适应播放支持

低风险 API 23+ 影响：中

当视频分辨率变化时（如 Mac 切换窗口大小或旋转屏幕），当前代码完全释放并重建 codec。使用 KEY_MAX_WIDTH + KEY_MAX_HEIGHT 可启用自适应播放，codec 内部处理分辨率变化，无需重建。

// 在 configureCodec() 中添加:
format.setInteger(MediaFormat.KEY_MAX_WIDTH, 3840)  // 最大支持 4K
format.setInteger(MediaFormat.KEY_MAX_HEIGHT, 2160)
// 之后分辨率变化只需 flush + 重新 start，无需 release/create

可行性：高。但需要确认设备解码器是否支持 FEATURE_AdaptivePlayback（大多数现代设备支持）。

预期收益：分辨率切换时无黑屏闪烁，切换耗时从 ~200ms 降到 ~20ms。

2.4 渲染质量优化

优化 9: SurfaceView 帧步调 (Choreographer)

中风险 API 24+ 影响：中

当前 onOutputBufferAvailable 中立即 releaseOutputBuffer(index, true)，不与显示 vsync 对齐。如果解码完成时恰好错过 vsync，帧要等下一个 vsync 周期（~16ms）才能显示。

有两种方案：

方案 A: releaseOutputBuffer timestamp（推荐）

// 使用渲染时间戳 API (API 21+)
// 传 0 表示立即渲染（当前行为）
// 传 System.nanoTime() 表示在下一个 vsync 渲染
override fun onOutputBufferAvailable(codec, index, info) {
    if (surface == null) {
        codec.releaseOutputBuffer(index, false)
        return
    }
    // 使用 0 = 立即渲染（保持当前行为，最安全）
    // 或使用 nanoTime 对齐 vsync（需要验证）
    codec.releaseOutputBuffer(index, 0L)
}

方案 B: Choreographer 帧回调

// 在主线程注册 Choreographer 回调，在 vsync 信号到达时
// 才 releaseOutputBuffer(index, true)
// 风险：需要跨线程协调，可能引入额外延迟

推荐：方案 A，传入 0L 保持立即渲染但明确使用 timestamp API。未来可基于 PTS 计算精确渲染时间。

7

推荐立即实施: Phase A（优化 1+4+8+9）

这四项优化全部是 configureCodec() 方法中添加几行 MediaFormat 键设置，以及 onOutputBufferAvailable 中一行 API 变更。

总代码改动量：~20 行。风险极低，不改变管线结构。预期效果：

  • 色彩：纠正 YUV→RGB 转换矩阵，消除色彩偏移（最明显感知提升）
  • 延迟：解码器更积极调度，减少 2-5ms 内部延迟
  • 流畅度：分辨率切换无黑屏，帧渲染更稳定
